# Provider Harness Expansion Work Items

Last updated: 2026-05-02 KST.

Goal: make `mos` provider execution reliable, parseable, and safe across Claude, Codex, Gemini, DeepSeek, Qwen, and local Ollama.

## P0 - Stabilize Adapter Contract

- Define `ProviderResult` schema with provider, role, status, mode, command, prompt path, output path, return code, timestamps, model, cost estimate, and parse errors.
- Write stdout and stderr to separate files; keep `output.md` as the human-readable merged view.
- Add `agents/<provider>/request.json` for HTTP providers.
- Add `provider_capabilities.json` generated by a probe command.
- Add one adapter registry in code instead of hard-coded `if agent == ...` branches.

Acceptance:

- `mos invoke <provider> --role reviewer` always creates prompt, command, output/result placeholders, and event log rows.
- Failed provider calls are represented as artifacts, not uncaught exceptions.

## P0 - Safe Execution Modes

- Map roles to modes: planner/reviewer/critic = read-only; executor/test-fixer = edit-capable; release/deploy = manual only.
- Forbid dangerous flags by policy: Claude permission bypass, Gemini `yolo`, broad shell execution, destructive commands.
- Add workspace-root guard so providers only run under `/home/user/workspaces/jaewon/myworld`.
- Add `--dry-run` to print commands and write artifacts without executing.

Acceptance:

- `mos invoke claude --role planner --execute` and `mos invoke gemini --role reviewer --execute` run in read-only mode by default.
- Any edit-capable mode requires an explicit flag.

## P1 - Claude Adapter

- Switch Claude execution to `--output-format json` by default.
- Add optional `--json-schema` for planner/reviewer outputs.
- Add model/effort/budget config fields.
- Capture and parse JSON into `result.yaml`.

Acceptance:

- Claude planner output produces parseable summary, decisions, risks, and next actions.

## P1 - Gemini Adapter

- Keep `--approval-mode plan` default.
- Add `--policy` support using generated MyWorld boundary policy files.
- Support `json` and `stream-json` capture.
- Record configured model and Gemini CLI version.

Acceptance:

- Gemini reviewer can run headlessly and produce a parseable review artifact.

## P1 - Codex Adapter

- Use `docs/cli_help.md` as the local CLI contract source for `codex exec`.
- Keep prompt/command artifact preparation always available.
- Add `codex exec` execution path for read-only reviewer roles with `--sandbox read-only --ask-for-approval never`.
- Keep edit-capable executor roles behind an explicit approval flag.
- Preserve access-gate failures such as `접근 거부` as provider result artifacts.
- Require changed files, commands run, tests run, and unresolved issues in result output.

Acceptance:

- `mos invoke codex --role reviewer --execute` writes command, output, and result artifacts even when local access is gated.
- `mos invoke codex --role executor --execute` is blocked or explicitly confirmed until edit-capable policy is implemented.

## P1 - HTTP Provider Adapter

- Implement OpenAI-compatible HTTP provider base class.
- Add DeepSeek config: `DEEPSEEK_API_KEY`, `https://api.deepseek.com`, default `deepseek-v4-flash`.
- Add Qwen config: `QWEN_API_KEY`, DashScope compatible base URL, configurable model.
- Normalize responses into `output.md` and `result.yaml`.
- Add retry/backoff and clear error classes for missing key, rate limit, model not found, and malformed JSON.

Acceptance:

- DeepSeek and Qwen can run reviewer/summarizer roles without local CLIs.

## P2 - Qwen CLI Adapter

- Detect optional `qwen` command.
- Reuse Gemini-style headless invocation where compatible: `qwen -p`.
- Probe approval flags and output-format support after install.
- Keep HTTP adapter as the default until local CLI behavior is verified.

Acceptance:

- If `qwen` is absent, `mos status` reports provider unavailable without failing.

## P2 - Ollama Health and Worker Hardening

- Add `mos providers probe ollama`.
- Check wrapper exists, server is running, model is pulled, and a smoke prompt returns valid JSON.
- Add model fallback: `qwen3:8b` -> `qwen3:1.7b` for cheap tasks; coding roles -> DeepSeek coder models.
- Persist latency and validation result per local worker call.

Acceptance:

- Local worker failures are visible in TUI and do not break run progression.

## P2 - TUI Integration

- Show provider availability, last run status, and execution mode.
- Add actions: prepare prompt, execute read-only, open output, mark accepted/rejected.
- Highlight unsafe modes before execution.

Acceptance:

- A user can run planner/reviewer/local-worker loop from TUI without inspecting folders manually.

## P3 - Memory Feedback Loop

- Convert accepted provider results into memory draft candidates.
- Store provider disagreement edges: Claude vs Gemini, hosted vs local, DeepSeek/Qwen cheap critique vs final decision.
- Track corrected local worker outputs for future tuning data.

Acceptance:

- `mos memory draft` includes provider outputs, accepted decisions, unresolved questions, and disagreement notes.
